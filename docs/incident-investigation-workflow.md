# Incident Investigation Workflow

## Overview

This document explains the incident investigation workflow used in the Azure Cloud Security Monitoring and Threat Visibility project.

The workflow is designed to demonstrate how an administrator or analyst can move from alert generation to triage, investigation, and response awareness using Microsoft Sentinel.

## Objective

Show how Microsoft Sentinel incidents can be reviewed, analyzed, and understood using:

* incident details
* related alerts
* entities
* evidence
* endpoint and identity context

## Project Investigation Scope

The investigation workflow in this project was designed primarily around identity-focused Microsoft Sentinel incidents generated from Microsoft Entra ID telemetry, including suspicious sign-in activity, password spray behavior, non-US sign-ins, and sign-ins to disabled accounts.

Where available, investigation context was supported by Defender-related telemetry and linked entities within Sentinel.

## Workflow Stages

### 1. Incident Creation

A Microsoft Sentinel analytics rule generates an alert based on matching log activity. Sentinel then creates an incident and places it in the incident queue.

### 2. Initial Triage

The first step is to review the incident summary and determine:

* what triggered the incident
* which user, device, or entity is involved
* whether the activity appears suspicious, expected, or requires more context

### 3. Incident Review

The incident is then reviewed in more detail by examining:

* incident severity
* associated alerts
* entities
* timestamps
* linked evidence

### 4. Entity Investigation

Investigation focuses on identifying the relevant users, endpoints, and activity involved in the incident.

For this project, that may include:

* sign-in activity from Microsoft Entra ID
* Defender-related telemetry from Microsoft Defender XDR
* device logon events
* evidence linked to alerts

### 5. Evidence and Context Review

The analyst reviews any available evidence to better understand:

* what happened
* where it happened
* which account or endpoint was involved
* whether the alert appears legitimate or malicious

Where available, the investigation graph is used to understand relationships between:

* incident-related entities

### 7. Administrative Awareness / Response

Once initial triage is complete, the project workflow includes response awareness through an email notification generated by the Sentinel playbook.

This project does not attempt automated containment or remediation. Instead, it demonstrates a lightweight operational response pattern:

* detect suspicious activity
* investigate the resulting incident
* notify the administrator through Sentinel automation

## Example Investigation Questions

During triage and review, useful questions include:

* Which rule created the incident?
* Which user or device is involved?
* Is this activity expected for the environment?
* Is there supporting evidence from endpoint or Defender-related telemetry?
* Does the incident require escalation or just documentation?

## Value of This Workflow

This workflow helps demonstrate:

* structured security operations thinking
* how to move from alert to incident
* how to use identity and endpoint telemetry together
* how Microsoft Sentinel supports triage and investigation
